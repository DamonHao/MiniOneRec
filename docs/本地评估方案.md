# MiniOneRec 本地模型评估方案（Apple Silicon MPS）

> 文档日期：2026-06-22  
> 当前状态：已完成。正式模型 `outputs/local_mps/sft-full/final_checkpoint` 已通过 1-beam、5-beam、8-sample 和 128-sample 真实 MPS constrained generation。  
> 核心结论：原有 8 卡 CUDA 评估流程已改造成 Apple Silicon 单进程流程。通过分块 `lm_head` 和固定 shape 的独立 constrained beam search，避开当前 MPSGraph/Transformers generation 的 shape 崩溃；128 条评估生成 640 个合法候选，HR@1/3/5 和 NDCG@1/3/5 均为 0。

## 1. 背景

仓库原有 `evaluate.sh` 面向 8 张 CUDA GPU，执行流程为：

```text
测试集
  → split.py 切分为 8 份
  → 8 个 CUDA 进程分别运行 evaluate.py
  → merge.py 合并预测
  → calc.py 计算 HR@K 和 NDCG@K
```

该流程不适合当前 Apple M5、24 GB 统一内存环境，主要原因如下：

- 硬编码 `CUDA_VISIBLE_DEVICES=0..7`，要求 8 个并行进程；
- `evaluate.py` 强制导入 `bitsandbytes`，本机 MPS 环境不使用该依赖；
- 模型以 `bfloat16` 和 `device_map="auto"` 加载，不符合本地单设备 `float32` 策略；
- Shell 使用 `python`，不符合项目统一通过 `uv run` 执行的规范；
- 临时目录不清理，进程失败后可能合并旧 JSON 或部分结果；
- `calc.py` 只打印指标，不生成便于追踪和比较的结构化指标文件；
- `CC` 在遇到真实目标后提前停止检查，会低估非法 SID 数量。

本地评估应保留原流程的核心语义：给定按时间排序的历史商品 SID，生成下一商品的 Top-K SID，并计算严格单真值的 HR@K、NDCG@K 和非法 SID 统计；同时移除 CUDA 多卡调度。

## 2. 目标

在 Apple Silicon 上建立单进程、可重复、可诊断的离线推荐评估链路：

```text
outputs/local_mps/sft-full/final_checkpoint + 仓库现有 test.csv + info.txt
  → 加载 checkpoint 自带 tokenizer
  → 根据商品目录构建 SID token trie
  → 单进程 constrained beam search
  → predictions.json
  → metrics.json（HR@K / NDCG@K / invalid SID）
```

完成标准：

- 使用 Python 3.11 arm64、`uv run` 和单个 MPS 进程；
- 不导入 CUDA、bitsandbytes、NCCL 或多进程评估组件；
- prompt 只包含历史 SID，不包含真实目标 SID；
- 使用 checkpoint 自带 tokenizer，按实际 tokenization 构建合法 SID trie；
- 每个测试样本返回指定数量的有序候选；
- 预测数量与读取的测试样本数量一致；
- 输出 HR@K、NDCG@K、非法 SID 数和非法 SID 比例；
- 预测与指标分别保存为 UTF-8 JSON；
- 至少完成 1 条测试样本的真实 MPS constrained generation；
- 指标、trie、输入校验和 Shell 参数均有自动化测试。

### 非目标

- 不复现论文规模和论文指标；
- 不支持 CUDA 多卡、分布式或多机评估；
- 不评估多样性、覆盖率、新颖性、公平性或在线业务收益；
- 不自动训练模型或生成 SID；
- 不静默回退到其他 checkpoint；
- 不把 SFT dry-run checkpoint 的指标当作正式推荐效果。

## 3. 要求

### 3.1 环境要求

- `.venv` 使用 uv-managed CPython 3.11 arm64；
- 依赖通过 `uv sync` 安装，Python 和 Shell 均通过 `uv run` 执行；
- PyTorch 必须支持 MPS，执行前确认 `torch.backends.mps.is_available()` 为 `True`；
- 设置 `PYTORCH_ENABLE_MPS_FALLBACK=1`，允许少量未实现算子回退 CPU；
- 默认 `TOKENIZERS_PARALLELISM=false`，避免无意义的 tokenizer 并行警告；
- 默认模型 dtype 为 `float32`，第一版不使用 bf16、量化或 8-bit 权重；
- 评估使用单进程，默认 batch size 为 1。

验证命令：

```bash
uv run python -c "import platform, torch; print(platform.machine()); print(torch.__version__); print(torch.backends.mps.is_available())"
```

预期包含：

```text
arm64
2.6.0
True
```

### 3.2 数据要求

评估需要同一套 SID 映射生成的三个输入：

```text
MODEL_PATH=outputs/local_mps/sft-full/final_checkpoint
INFO_FILE=data/Amazon/info/Industrial_and_Scientific_5_2016-10-2018-11.txt
TEST_FILE=data/Amazon/test/Industrial_and_Scientific_5_2016-10-2018-11.csv
```

必须满足：

- `info.txt` 非空，每行第一列是合法完整 SID；
- `test.csv` 的 `history_item_sid` 是字符串列表，`item_sid` 是单个真实目标；
- 所有历史 SID 和目标 SID 都存在于 `info.txt`；
- checkpoint 中的 tokenizer 与训练时注册的 SID token 一致；
- 当前 `sft-full` checkpoint 使用仓库现有 Industrial SID/index 训练，因此必须搭配上述 `data/Amazon` 的 `info/test`；
- 新 `data/local_mps` 数据只能搭配未来使用新 SID 训练的 checkpoint，不能与当前模型混用。

### 3.3 模型与生成要求

本机默认参数：

| 参数 | 默认值 | 说明 |
| --- | ---: | --- |
| device | `mps` | 显式要求 MPS，不自动假装成功 |
| dtype | `float32` | 本机第一版稳定优先 |
| batch size | 1 | 控制统一内存占用 |
| beams | 5 | 对应 Top-5 候选 |
| max input tokens | 256 | 与本地训练长度对齐 |
| max new tokens | 16 | 覆盖 SID、换行和 EOS |
| Top-K | 1、3、5 | 不得大于 beams |
| test samples | 128 | smoke test 先使用 1 或 8 |
| seed | 42 | 保证可重复 |

tokenizer 必须使用左侧 padding 和左侧 truncation。长序列截断时应保留最近历史和末尾 `### Response:`，不能从右侧删除真正参与生成的 prompt 后缀。

## 4. 假设

1. “评估模型”指严格的 next-item Top-K 离线评估，不代表在线推荐效果。
2. 每个测试样本只有一个真实下一商品，因此本方案的 HR@K 数值等价于 Recall@K。
3. 指标通过 SID 字符串精确匹配，不把同类商品或语义相似商品视为命中。
4. SID 可能包含三层或更多层，约束逻辑必须根据 tokenizer 实际编码动态构建，不能假定固定 token 数。
5. 训练目标是 `SID + 换行 + EOS`，评估约束序列也应包含换行，以保持候选序列打分与训练目标一致。
6. `num_beams` 同时表示 Beam Search 宽度和每个样本返回的候选数量。
7. 若指定 MPS 但设备不可用，程序应立即失败；只有用户显式指定 `device=cpu` 时才允许 CPU 评估。
8. 现有 SID 的 16 个碰撞可以用于工程 smoke test，但不能作为正式推荐效果基线。
9. 当前 MPSGraph 对扩展后超宽 `lm_head` 的限制属于运行时兼容问题，不应通过关闭约束、缩小商品目录或静默改用 CPU 掩盖。

## 5. 实现方案

### 5.1 本地评估入口

本地代码放在：

```text
scripts/local/evaluate_mps.py
scripts/local/run_evaluate.sh
```

保持原有 CUDA 入口不变：

```text
evaluate.sh
evaluate.py
split.py
merge.py
calc.py
```

这样可以避免 Apple Silicon 改造破坏仓库原有多卡流程。

本地 Shell 入口负责：

1. 设置 `UV_CACHE_DIR`、MPS fallback 和 tokenizer 环境变量；
2. 校验 checkpoint、测试 CSV 和 info 文件存在；
3. 创建结果目录；
4. 使用 `uv run python` 启动单进程评估；
5. 允许通过环境变量覆盖模型、数据和生成参数。

### 5.2 输入解析和 prompt

测试 CSV 只读取：

```text
history_item_sid
item_sid
```

`history_item_sid` 使用安全的字面量解析，不直接调用 `eval()`。生成 prompt 与训练核心任务保持一致：

```text
### Instruction:
Can you predict the next possible item that the user may expect?

### User Input:
Can you predict the next possible item the user may expect, given the following chronological interaction history: <sid_1>, <sid_2>, ...

### Response:
```

真实 `item_sid` 只保存在结果记录中用于指标计算，不能拼入模型输入。

### 5.3 SID 约束生成

读取 `info.txt` 第一列得到完整商品 SID。每个候选按以下方式编码：

```text
tokenizer.encode(SID + "\n", add_special_tokens=False)
```

所有编码序列组成 token trie：

- 空前缀只允许合法 SID 的首 token；
- 中间前缀只允许 trie 中存在的下一 token；
- 完整 `SID + 换行` 后只允许 EOS；
- 不在 trie 中的前缀立即报错，不能继续生成非法候选；
- trie 不依赖 SID 固定为三层。

每个 batch 单独建立生成约束闭包，只截取 prompt 后的新生成 token，避免原 `ConstrainedLogitsProcessor` 使用跨 batch 可变计数器造成状态泄漏。

### 5.4 MPS 超宽输出头兼容方案

当前 dry-run checkpoint 的关键 shape：

```text
hidden states: [batch, sequence, 896]
lm_head weight: [152225, 896]
```

真实验证结果：

- Qwen Transformer 主体在 MPS 上前向成功；
- 一次性执行 `[*, 896] × [896, 152225]` 时，MPSGraph 报 `mps_matmul incompatible dimensions` 并触发 LLVM abort；
- greedy 和 Beam Search 都会失败，因此问题不是 beams 数量；
- 同一 checkpoint 在 CPU 上能完成 constrained generation；
- 设置 `PYTORCH_MPS_PREFER_METAL=1` 仍不能修复；
- 将 `lm_head.weight` 沿词表维度按 8,192 行分块，分别执行 `F.linear` 后拼接，真实 MPS 前向成功，shape 为 `[1, 1, 152225]` 且结果全部有限。
- 分块输出头接入后，Transformers `generate()` 在 greedy、关闭 KV cache 和缩小 chunk/input 后仍触发相同 LLVM abort；
- 同一完整 prompt 的 Transformer 主体与分块输出头分别执行均成功，因此剩余故障位于通用 `generate()` 的 MPS 生成编排路径。

本地入口已经采用数学等价的分块输出投影：

```text
hidden states
  → chunk 1 linear: vocab [0:8192]
  → chunk 2 linear: vocab [8192:16384]
  → ...
  → concatenate logits on vocabulary dimension
```

实现要求：

- 只在 MPS 评估路径替换 `lm_head` 前向，不修改或重新保存 checkpoint；
- 复用原 `lm_head` 的 weight/bias，不复制完整参数；
- CPU 上用自动化测试证明分块投影与原 `nn.Linear` 数值等价；
- chunk size 作为参数暴露，本机默认 8,192；
- 接入后验证 greedy、5-beam、1-sample 和 8-sample MPS 生成；
- 若分块后仍失败，停止评估并记录错误，不能静默回退 CPU。

为避开当前 Transformers `generate()` 的 MPS 路径，本地评估改用独立 constrained beam search：

1. 每个样本从空 completion 和分数 0 开始；
2. 每一步将 prompt 与当前 beam completion 拼接后执行完整模型前向；
3. 显式设置 `use_cache=False`，并只计算最后一个位置的 logits；
4. 根据 SID trie 取出每个 beam 当前允许的 token；
5. 累加 log-softmax 分数，保留全局最高的 `num_beams` 条路径；
6. 生成 EOS 的路径标记为完成，不再增加分数；
7. 所有 beam 完成后返回排序候选；超过 `max_new_tokens` 仍未完成则明确失败。

MPSGraph 对同一前向图的动态 shape 复用不稳定，因此搜索还必须保证每一步模型输入 shape 恒定：

- sequence length 固定为 `prompt_length + max_new_tokens`；
- prompt/completion 左侧使用 pad token 补齐，attention mask 对 padding 置 0；
- 模型 batch 维固定为 `num_beams`；首步或完成 beam 导致活跃路径不足时，只为模型前向复制占位输入，不把占位 logits 加入候选；
- 每一步只消费真实活跃 beam 对应的 logits。

该搜索保持 `length_penalty=0` 的累积 log-probability 排序语义。由于默认评估 batch size 为 1，第一版按样本顺序执行，不依赖 Transformers GenerationMixin、KV cache 或多进程。

### 5.5 指标计算

默认计算：

```text
HR@1、HR@3、HR@5
NDCG@1、NDCG@3、NDCG@5
invalid_sid_count
invalid_sid_rate
```

若真实商品位于第 `rank` 名：

```text
HR@K   = 1                         if rank <= K else 0
NDCG@K = 1 / log2(rank + 1)        if rank <= K else 0
```

最终对全部测试样本取平均。非法 SID 检查必须遍历全部候选，不能在命中真实目标后提前退出。

预测文件结构：

```json
{
  "history_item_sid": ["<a_1><b_2><c_3>"],
  "target_sid": "<a_2><b_3><c_4>",
  "predictions": ["<a_4><b_5><c_6>", "<a_2><b_3><c_4>"]
}
```

指标文件还应记录模型路径、设备、dtype、beams、batch size、最大输入/输出长度和随机种子，便于复现实验。

### 5.6 测试优先的实施顺序

应按以下顺序实施和验证：

1. SID trie 合法路径和 EOS 测试；
2. HR/NDCG 排名及非法 SID 全量检查测试；
3. 空数据、非法 Top-K、缺失字段和未知 SID 测试；
4. tokenizer 左侧 padding/truncation 测试；
5. Shell 默认参数和环境变量覆盖测试；
6. 分块 `lm_head` 与原始 Linear 的数值等价测试；
7. 使用确定性 fake model 验证独立 constrained beam search 的路径、排序和每步固定 shape；
8. Python 编译和 Shell 语法检查；
9. CPU 1-sample constrained generation，用于隔离数据/模型问题；
10. MPS 1-sample greedy generation；
11. MPS 1-sample 5-beam generation；
12. MPS 8-sample smoke test；
13. 最终 128-sample 本地评估。

### 5.8 停止条件与降级顺序

出现以下情况立即停止：

- MPS LLVM abort、OOM 或系统 memory pressure 进入红色；
- tokenizer/model vocab 不一致；
- 历史或目标 SID 不在商品目录；
- 预测样本数与测试样本数不一致；
- constrained generation 出现非法前缀；
- 指标出现 NaN/Inf；
- checkpoint 无法重载。

若内存不足，按以下顺序降级：

1. 保持 batch size 为 1；
2. 将 beams 从 5 降到 3，再降到 1，仅用于定位问题；
3. 将 max input tokens 从 256 降到 192 或 128；
4. 将 smoke 样本数降到 1；
5. 调小 `lm_head` chunk size；
6. 只有用户显式要求时才运行 CPU 对照，不能把 CPU 结果标记为 MPS 通过。

### 6.3 最终判断

- **单进程本地评估已跑通。** 数据读取、prompt、动态 SID trie、分块输出头、固定-shape beam search、结构化指标和 Shell 入口均已完成验证。
- **MPS 兼容问题已规避。** 原始超宽 `lm_head` 和 Transformers `generate()` 仍会触发底层 abort，但本地入口不再执行这两条不稳定路径。
- **当前模型的离线命中率为 0。** 128 条样本上 HR/NDCG@1/3/5 全部为 0；候选均合法，因此该结果应归因于模型效果或训练配置，而不是非法生成。
- **正式评估对象已固定。** 本阶段评估 `outputs/local_mps/sft-full/final_checkpoint`，不等待 RL checkpoint，也不混用 `data/local_mps` 的未来 SID 数据。
- **CPU 结果只用于隔离问题。** CPU constrained generation 成功证明 checkpoint、测试数据和约束逻辑一致，但不能替代 MPS 验收。
