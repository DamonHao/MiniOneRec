# MiniOneRec 本地模型评估方案（原评估协议的 Apple Silicon MPS 适配）

> 文档日期：2026-06-22  
> 当前状态：代码协议对齐已完成；原数据/约束/JSON/指标测试及 CPU 5/50-beam 候选顺序 parity 已通过。MPS 50-beam 和完整 4,533 条评估因当前外部 Metal 执行额度限制尚未运行。  
> 核心结论：本地评估只能修改设备、dtype、模型输出投影和 MPS 不兼容的生成执行方式；数据管线、prompt token、约束空间、Beam Search 排序语义、正式参数、输出格式和指标定义必须以原 `evaluate.py + data.py + calc.py` 为基线。

## 1. 背景

仓库原始离线评估由以下文件共同定义，不能只看 `evaluate.py`：

```text
evaluate.sh
  → split.py
  → evaluate.py
       → EvalSidDataset（data.py）
       → ConstrainedLogitsProcessor（LogitProcessor.py）
  → merge.py
  → calc.py
```

其中：

- `evaluate.py` 负责加载模型、构建合法 SID 前缀约束并调用 Beam Search；
- `EvalSidDataset` 负责生成 Instruction、User Input、历史 SID 和目标 SID；
- `ConstrainedLogitsProcessor` 负责把生成限制在商品 SID 目录内；
- `calc.py` 定义 HR@K、NDCG@K 和 CC 的最终计算行为；
- `evaluate.sh` 负责 8 卡拆分、合并和参数调度。

Apple Silicon 无法直接运行原流程，因为原入口硬编码 8 张 CUDA GPU、`bfloat16`、`device_map="auto"` 和 CUDA 相关依赖。本地方案需要把执行方式改成单进程 MPS，但不能因此重新定义评估任务。

### 1.1 已定位的行为偏差

现有 `scripts/local/evaluate_mps.py` 绕过 `EvalSidDataset`，自行读取 CSV 并手工实现了 `build_prompt()`。该逻辑不是 MPS 必要改造，而是额外重写了原数据管线。

对同一测试样本和正式 checkpoint 实测：

```text
原 EvalSidDataset input token 数：121
当前 build_prompt input token 数：120
token IDs 完全一致：False
首次差异位置：26
```

直接原因之一是原 Instruction 的 `request.` 后保留了一个空格，而手工版本删除了该空格。模型实际输入已经变化，因此当前结果不能视为原评估的 MPS 等价结果。

除 prompt 外，当前实现还存在以下协议差异：

| 项目 | 原评估行为 | 当前 MPS 行为 | 判断 |
| --- | --- | --- | --- |
| 数据管线 | `EvalSidDataset` | 已恢复直接复用 | 已对齐 |
| prompt/tokenizer | `data.py` 原始拼接和 `Tokenizer.encode` | 已删除正式路径手工 prompt | 已对齐 |
| 输入截断 | 测试分支直接返回完整 tokens | 不再额外截断 | 已对齐 |
| 正式测试样本 | 完整 test split | 默认 `MAX_SAMPLES=-1` | 已对齐 |
| Beam 数 | 50 | 默认 50 | 已对齐 |
| 最大生成长度 | 256 | 默认 256 | 已对齐 |
| 指标 K | 1、3、5、10、20、50 | 已恢复全部 K | 已对齐 |
| 预测 JSON | `input/output/predict` | 已恢复原字段 | 已对齐 |
| 指标实现 | 原 `calc.py` | Shell 调用原 `calc.py`，结构化结果通过 fixture | 已对齐 |
| CC | 按原 `calc.py` 行为 | 已恢复命中后停止检查 | 已对齐 |
| 设备 | CUDA | MPS | 允许修改 |
| dtype | bf16 | float32 | MPS 必要修改 |
| 模型输出头 | 单次 Linear | 分块 Linear | MPS 必要修改，需证明数值等价 |
| 生成执行 | Transformers Beam Search | 固定-shape 自定义搜索 | 仅在排序行为等价后允许 |

此前完成的 128-sample、5-beam MPS 结果只能证明 MPS 工程链路可运行，不能作为原评估协议下的正式指标。

## 2. 目标

在 Apple Silicon MPS 上运行与原评估协议行为一致的离线 Top-K 评估：

```text
原 EvalSidDataset 输入
  → 原 SID 前缀约束
  → 与原 Beam Search 等价的 MPS 生成
  → 原 input/output/predict JSON
  → 原 calc.py 指标
```

正式模型和配套数据固定为：

```text
MODEL_PATH=outputs/local_mps/sft-full/final_checkpoint
INFO_FILE=data/Amazon/info/Industrial_and_Scientific_5_2016-10-2018-11.txt
TEST_FILE=data/Amazon/test/Industrial_and_Scientific_5_2016-10-2018-11.csv
CATEGORY=Industrial_and_Scientific
```

完成标准：

- 相同样本经原管线和 MPS 管线得到完全一致的 `input_ids` 和 `attention_mask`；
- SID token 序列及每一步允许 token 集合与原约束逻辑一致；
- 使用相同模型、dtype 和 CPU 设备时，原生成与适配生成的候选 SID 及顺序一致；
- 正式参数恢复为 50 beams、256 max new tokens 和完整测试集；
- 输出 JSON 恢复为原 `input/output/predict` 格式；
- 正式 HR/NDCG/CC 由原 `calc.py` 计算，或由通过逐项等价测试的共享函数计算；
- 单进程 MPS 评估完成，不触发 LLVM abort、OOM 或静默 CPU 回退；
- 原 CUDA 文件保持不变。

### 非目标

- 不修正原 prompt 文案、空格或语法；
- 不替换原 `EvalSidDataset` 的字段和输出结构；
- 不改变原 Beam Search 的排名定义；
- 不把 5-beam、128-sample smoke 参数作为正式参数；
- 不修复或重新解释原 `calc.py` 的 CC 行为；
- 不复现论文硬件规模或论文指标；
- 不修改训练 checkpoint。

## 3. 要求

### 3.1 原评估行为是唯一基线

行为优先级如下：

1. `data.py` 的 `EvalSidDataset`：定义模型输入；
2. `evaluate.py`：定义 SID 约束和生成参数；
3. `calc.py`：定义正式指标；
4. `evaluate.sh`：定义正式类别和调度参数。

`evaluate.sh` 当前传入但 `evaluate.py.main()` 不接收的 `temperature` 和 `guidance_scale` 属于入口缺陷，不构成有效评估语义，不要求在 MPS 版本中模拟。

当前 checkpoint 自带采样型 `generation_config.json`。Transformers 4.57 会把其中的 `do_sample`、`temperature`、`top_k/top_p` 和 `repetition_penalty` 注入新建配置；这与原脚本标记的 `STANDARD MODE` 及 `GenerationConfig` 默认确定性 Beam Search 本意冲突。协议 parity 以原代码本意为准：禁用 model generation defaults，不继承 checkpoint 采样参数。

### 3.2 正式评估参数

正式 MPS 评估应与原 `evaluate.sh/evaluate.py` 对齐：

| 参数 | 正式值 | 说明 |
| --- | ---: | --- |
| category | `Industrial_and_Scientific` | 与模型和数据一致 |
| seed | 42 | 与原默认一致 |
| batch size | 8 | 对外协议保持一致；内部可按样本执行以适配 MPS |
| num beams | 50 | 同时返回 50 个候选 |
| max new tokens | 256 | 与原 Shell 一致 |
| length penalty | 0.0 | 累积 log probability 排序 |
| test samples | 完整 test split | 当前文件共 4,533 条 |
| Top-K | 1、3、5、10、20、50 | 与 `calc.py` 一致 |

Smoke test 可以显式使用较小参数，但必须在命令和结果中标记为 smoke，不能覆盖正式默认值。

### 3.3 数据与输出格式

MPS 版本必须直接复用 `EvalSidDataset`：

```python
EvalSidDataset(
    train_file=test_data_path,
    tokenizer=tokenizer,
    max_len=2560,
    category=category_text,
    test=True,
    K=K,
    seed=seed,
)
```

注意：当前 `EvalSidDataset.pre()` 的测试分支直接返回构造后的 tokens，不执行额外截断。MPS 版本不得再增加 `max_input_tokens=256` 截断。

正式预测 JSON 恢复为原结构：

```json
{
  "input": "Can you predict ... chronological interaction history: <sid_1>, <sid_2>",
  "output": "<target_sid>\n",
  "predict": ["<sid_1>", "<sid_2>"]
}
```

允许另行输出运行元数据 JSON，但不能替代或修改原预测文件。

### 3.4 MPS 环境要求

- Python 固定为 3.11 arm64；
- 所有命令通过 `uv run` 执行；
- 单进程使用 `mps`，不使用 CUDA、NCCL、DeepSpeed 或 bitsandbytes；
- dtype 使用 `float32`；
- 设置 `PYTORCH_ENABLE_MPS_FALLBACK=1`，但不得静默把整个模型移到 CPU；
- checkpoint tokenizer 和 model vocab 必须一致；
- MPS 不可用时立即失败。

## 4. 假设

1. 评估协议一致指输入构造、候选空间、搜索排序、输出和指标定义一致；不要求 CUDA bf16 与 MPS float32 的浮点位级一致。
2. 生成算法的等价性应先在相同 CPU/dtype 环境比较，排除设备浮点误差。
3. 分块 `lm_head` 是同一 Linear 的数学分块，不改变权重和 logits 定义。
4. 若 Transformers `generate()` 无法在 MPS 上运行，可以使用自定义固定-shape搜索，但必须通过 CPU 候选顺序对照。
5. 原商品 SID 当前为固定三层，但约束行为仍以原 `info.txt` tokenization 为准，不能凭格式假设重新定义。
6. 原 `calc.py` 的 CC 可能低估非法候选，但正式 CC 必须保持原行为；完整非法 SID 数可以使用新名称作为附加诊断指标。
7. 当前 128 条、5-beam、HR/NDCG 为 0 的结果不纳入正式效果结论，直到协议一致性验收完成。

## 5. 实施规划

### 5.1 阶段 A：建立原评估黄金基线

先增加只读对照测试，不修改生产逻辑：

1. 使用正式 checkpoint tokenizer 和原 `EvalSidDataset` 生成黄金输入；
2. 对完整 test split 或至少覆盖短/长历史的固定样本，保存或即时比较 `input_ids`、`attention_mask`；
3. 固定原 `input/output` 字段格式；
4. 从原 `evaluate.py` 的 `hash_dict` 逻辑生成约束黄金结果；
5. 使用人工预测 fixture 记录原 `calc.py` 的 HR/NDCG/CC 输出。

阶段验收：测试必须能捕获当前手工 `build_prompt()` 的 121 vs 120 token 差异。

### 5.2 阶段 B：恢复原数据管线

调整 MPS 入口：

- 删除手工 `build_prompt()` 作为正式输入来源；
- 删除正式路径中的独立 CSV prompt 重建；
- 直接实例化原 `EvalSidDataset`；
- 使用 `val_dataset[i]` 获取模型输入；
- 使用 `val_dataset.get_all()` 获取原 `input/output/dedup` 记录；
- 预测完成后仅增加 `predict`，并与原逻辑一样删除 `dedup`；
- 不增加额外输入截断。

阶段验收：代表样本及完整数据的 token IDs、attention mask、目标和 JSON 基础字段与原管线一致。

### 5.3 阶段 C：恢复原 SID 约束

优先复用或抽取原 `evaluate.py` 的约束构建逻辑：

- 保留 `### Response:\n{SID}\n` 的原 tokenization；
- 保留当前模型对应的 `prefix_index` 行为；
- 保留 EOS 处理；
- 对每个可能前缀比较原 `hash_dict` 与 MPS trie 的允许 token 集合；
- 若 trie 不能证明等价，直接复用原约束数据结构。

阶段验收：所有合法 SID 路径及终止位置的允许 token 集合完全一致。

### 5.4 阶段 D：验证生成算法等价性

原 Transformers Beam Search 是排序行为基线。当前固定-shape自定义搜索只能在以下测试通过后保留：

1. 使用 deterministic fake logits 覆盖分支、EOS、完成候选和分数并列；
2. 在 CPU、相同 float32 模型上，自定义搜索关闭 MPS 固定-shape padding，对固定样本分别运行原生成和自定义生成；
3. 先比较 1 beam，再比较 5 beams，最后比较 50 beams；
4. 候选 SID 集合和顺序必须一致；
5. 专门覆盖不同完成长度，确认 finished beam 管理与 Transformers Beam Search 一致；
6. 证明 `length_penalty=0.0`、`num_return_sequences=num_beams` 行为一致。

若候选或顺序不一致，必须修正自定义搜索，不能以“候选都合法”作为通过标准。

### 5.5 阶段 E：隔离 MPS 必要改造

仅保留以下 MPS 差异：

- 模型显式加载到单个 `mps` 设备；
- dtype 改为 `float32`；
- 不导入 bitsandbytes；
- 超宽 `lm_head` 使用分块 Linear；
- 为规避 MPSGraph 动态 shape 问题，模型前向使用固定 batch/sequence shape；
- 固定 shape 时保持有效 prompt/completion 左对齐，只在右侧 padding；显式传递 attention mask 和 position IDs，并通过 tensor `logits_to_keep` 只计算当前最后一个有效 token 的 logits，避免 padding 改变原 token 位置；
- 必要时关闭 KV cache；
- Shell 使用 `uv run`，不进行 8 卡拆分。

分块 Linear 必须在 CPU 上与原 Linear 通过数值等价测试；固定 shape 右侧 padding、attention mask、position IDs 和选定 logits 位置必须通过 CPU Beam Search parity，证明不改变有效 token 语义。

CPU parity 路径不启用固定 shape；固定 shape 仅用于 MPS。MPS 与 CPU 因矩阵 shape 和设备实现产生的微小浮点差异允许导致边界候选变化，但搜索分数定义、约束和候选排序规则必须相同。

### 5.6 阶段 F：恢复原输出和指标

- 正式预测写为 `input/output/predict`；
- 正式运行完成后调用原 `calc.py`；
- 报告 HR/NDCG@1、3、5、10、20、50；
- 保留原 CC 定义；
- 如需完整非法 SID 统计，使用 `invalid_sid_count_full` 等新字段作为附加诊断，不替代 CC；
- 结构化 metrics JSON 必须通过 fixture 与 `calc.py` 输出逐项对照。

### 5.7 阶段 G：分级验收

严格按以下顺序执行：

1. 原数据管线 token parity；
2. SID 约束 parity；
3. 指标 parity；
4. CPU 1-sample / 1-beam generation parity；
5. CPU 1-sample / 5-beam generation parity；
6. CPU 固定样本 / 50-beam generation parity；
7. MPS 1-sample / 1-beam smoke；
8. MPS 1-sample / 50-beam smoke；
9. MPS 8-sample / 50-beam smoke；
10. 完整 4,533-sample / 50-beam 正式评估；
11. 使用原 `calc.py` 计算最终指标。

任何 parity 阶段失败，都不得进入正式全量评估。

### 5.8 目标运行方式

正式运行默认不缩减样本和 beams：

```bash
MODEL_PATH=outputs/local_mps/sft-full/final_checkpoint \
INFO_FILE=data/Amazon/info/Industrial_and_Scientific_5_2016-10-2018-11.txt \
TEST_FILE=data/Amazon/test/Industrial_and_Scientific_5_2016-10-2018-11.csv \
CATEGORY=Industrial_and_Scientific \
NUM_BEAMS=50 \
MAX_NEW_TOKENS=256 \
MAX_SAMPLES=-1 \
uv run bash scripts/local/run_evaluate.sh
```

Smoke test 必须显式声明缩减参数，并写入独立结果路径：

```bash
NUM_BEAMS=5 \
MAX_NEW_TOKENS=16 \
MAX_SAMPLES=8 \
PREDICTIONS_PATH=results/local_mps/smoke-predictions.json \
uv run bash scripts/local/run_evaluate.sh
```

### 5.9 停止条件

出现以下任一情况立即停止：

- prompt token IDs 与原 `EvalSidDataset` 不一致；
- 合法前缀的允许 token 集合不一致；
- CPU 候选 SID 或顺序不一致；
- JSON 字段或目标格式不一致；
- HR/NDCG/CC 与原 `calc.py` 不一致；
- MPS LLVM abort、OOM 或整体回退 CPU；
- 预测数量不等于测试样本数量；
- 50-beam 结果不足 50 个候选。
