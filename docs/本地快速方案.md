# MiniOneRec 本地快速 SFT 方案（复用现有 SID）

> 文档日期：2026-06-21  
> 当前状态：MPS 崩溃已定位并修复；修复后的完整 8-sample dry run 待本机复验。  
> 核心结论：直接复用仓库现有 Industrial 数据、item metadata 和 SID，可跳过原始数据处理、文本向量、SID 构建及 finalize，用 0.5B 模型和小样本在 Apple M5 上快速验证 SFT 训练链路。

## 1. 背景

完整本地方案需要执行 item embedding、SID 构建、格式转换、SFT、RL 和离线评估。首次本地验证若同时处理所有阶段，问题定位范围过大，也会增加模型下载、CPU 聚类和 MPS 训练耗时。

仓库已包含一组可以直接用于 SFT 的一致数据：

```text
data/Amazon/index/Industrial_and_Scientific.index.json
data/Amazon/index/Industrial_and_Scientific.item.json
data/Amazon/train/Industrial_and_Scientific_5_2016-10-2018-11.csv
data/Amazon/valid/Industrial_and_Scientific_5_2016-10-2018-11.csv
```

静态核验结果：

| 项目 | 结果 |
| --- | ---: |
| item 数 | 3,686 |
| index 数 | 3,686 |
| SID 层数 | 3 |
| SID token 数 | 560 |
| 唯一完整 SID 数 | 3,670 |
| 完整 SID 碰撞数 | 16 |
| train 行数 | 36,259 |
| valid 行数 | 4,532 |
| train + valid 与 index 一致行数 | 40,791 / 40,791 |

现有 SID 碰撞不会阻止 tokenizer 扩展、loss 反向传播和 checkpoint 保存，因此可以用于 SFT smoke test；但碰撞会造成部分 item 无法通过 SID 唯一区分，不能把本方案的训练结果当作正式推荐效果基线。

## 2. 目标

在 Apple M5、24 GB 统一内存环境中，用最小成本完成阶段 G 的功能性 SFT：

```text
现有 CSV + 现有 SID/index + 现有 item metadata
  → 加载 Qwen2.5-0.5B-Instruct
  → 注册 SID token
  → MPS 小样本 SFT
  → 保存并重新加载 final checkpoint
```

完成标准：

- 使用 Python 3.11 arm64 和单进程 MPS；
- 不执行 embedding、RQ-KMeans、`bootstrap` 或 `finalize`；
- 成功向 tokenizer 注册现有 560 个 SID token；
- 至少完成 1 个 epoch，训练 loss 和验证 loss 为有限值；
- 生成 `outputs/local_mps/sft-smoke/final_checkpoint`；
- model、tokenizer 和新增 SID token 均可从 checkpoint 重新加载；
- 训练过程中不发生 MPS OOM、进程强制终止或 checkpoint 损坏。

### 非目标

- 不解决现有 SID 的 16 个碰撞；
- 不复现论文 SFT 规模或指标；
- 不执行 RL 和离线推荐指标评估；
- 不使用完整 36,259 条训练数据；
- 不使用 CUDA、NCCL、DeepSpeed、bitsandbytes 或多进程训练；
- 不用本次 smoke checkpoint 替代正式全流程 checkpoint。

## 3. 要求

### 3.1 环境要求

- `.venv` 必须为 uv-managed CPython 3.11 arm64；
- PyTorch 2.6.0 的 `mps.is_available()` 必须为 `True`；
- 依赖通过 `uv sync` 安装，所有命令通过 `uv run` 执行；
- 允许设置 `PYTORCH_ENABLE_MPS_FALLBACK=1`；
- 开始训练前关闭明显占用统一内存的应用；
- 可用磁盘不得低于 15 GiB。

### 3.2 数据要求

本方案直接使用仓库现有文件，不读取 `data/local_mps/staging`，也不执行 `finalize`。

必须同时满足：

- index 和 item JSON 都有 3,686 个 key；
- CSV 中的 `history_item_id`、`item_id` 都能在 index 中找到；
- CSV 中的 `history_item_sid`、`item_sid` 与 index 一致；
- index、item metadata 和 CSV 必须作为同一组数据使用，不能混用新旧 SID。

### 3.3 模型与训练要求

- 基础模型：`Qwen/Qwen2.5-0.5B-Instruct`；
- 从预训练权重加载，`train_from_scratch=False`；
- 第一轮使用 `float32`，不启用 bf16、fp16 或 8-bit optimizer；
- 单设备 MPS，`micro_batch_size=1`；
- 最大长度使用 192，避免现有 prompt 在 64/128 下被过度左截断；
- 第一轮只训练新增 SID 的输入/output embedding，冻结 Transformer 主体；
- optimizer 使用 `adamw_torch`，weight decay 保持 0，避免被 mask 的旧 token 行发生权重衰减。

## 4. 假设

1. 本方案的“完成 SFT”指工程链路通过，不代表模型达到可用推荐效果。
2. 允许接受现有 16 个完整 SID 碰撞，只要数据与 index 保持一致。
3. `sample=32` 会分别作用于 `SidSFTDataset`、`SidItemFeatDataset` 和 `FusionSeqRecDataset`，因此训练集约为 96 条，不是总计 32 条；验证集约为 32 条。
4. Qwen tokenizer 可以通过 `add_tokens` 注册 SID token，模型 input embedding 和 output head 会随 tokenizer 一起 resize。
5. Qwen input embedding 和 output head 不应假定权重绑定；冻结模式必须同时处理两者。
6. 首次运行可以联网下载 Qwen 模型；若网络不可用，需提前准备本地模型目录。
7. 实际耗时与峰值内存必须通过实跑记录，方案不预设完成时间。

## 5. 实现方案

### 5.1 快速路径

本方案跳过完整流程的阶段 2、3、5.2 数据生成和 finalize：

```text

  → 新建 sft_mps.py
```

原因是快速方案不生成新 SID，直接使用 `data/Amazon` 中已经匹配的 CSV、index 和 item metadata。

### 5.3 新建独立的 `sft_mps.py`

**状态：已完成。** 现有 `sft.py` 未修改。项目根目录下已新增 `sft_mps.py`，独立实现 Apple Silicon SFT 入口，并复用 `data.py` 中的数据集类。`sft_mps.py` 不从 `sft.py` 导入，避免触发 bitsandbytes 等 CUDA 依赖。

#### 5.3.1 隔离 CUDA-only 依赖

- `sft_mps.py` 不导入 bitsandbytes、DeepSpeed 或 CUDA-only 模块；
- 不使用 `torchrun`、NCCL 或 DeepSpeed；
- 保持单进程执行。

#### 5.3.2 增加设备和 dtype 参数

新增参数：

```text
device=mps
dtype=float32
gradient_checkpointing=False
```

设备选择规则：

1. 显式传入 `mps` 时，若 MPS 不可用立即失败；
2. `auto` 时优先 MPS，其次 CUDA，最后 CPU；
3. MPS 模型使用 `torch.float32`；
4. 不使用 `device_map="auto"`，模型显式移动到单一设备。

#### 5.3.3 修复 SID token 冻结训练

原始 `sft.py` 的 `freeze_LLM=True` 分支使用了未定义的 `original_vocab_size`，且只处理 input embedding；独立 `sft_mps.py` 已按以下方式修复：

1. 在 `tokenizer.add_tokens` 前记录 `original_vocab_size`；
2. 注册 560 个 SID token 后 resize input embedding 和 output head；
3. 冻结 Transformer 主体；
4. 同时允许 input embedding 和 output head 计算梯度；
5. 对旧词表行注册 gradient mask，只保留新增 SID token 行的梯度；
6. 若 input/output 权重实际绑定，只注册一次 hook；
7. 训练前打印新增 token 数、可训练参数和总参数。

该模式的目标是最小成本验证 SID token 能被训练，不等同于完整参数 SFT。

#### 5.3.4 调整 TrainingArguments

MPS smoke 配置：

```text
bf16=False
fp16=False
optim=adamw_torch
weight_decay=0.0
gradient_checkpointing=False
dataloader_pin_memory=False
eval_strategy=no
save_strategy=no
save_total_limit=1
load_best_model_at_end=False
report_to=none
```

训练开始前设置 `model.config.use_cache=False`。在当前 PyTorch 2.6/Qwen/MPS 组合中，Trainer 的 gradient-checkpointing 路径和模型更新后的 MPS `no_grad` 验证路径均可能触发 `mps_matmul` shape/LLVM 崩溃。因此 MPS 下禁用 gradient checkpointing、epoch eval 和中间 checkpoint；训练结束后把模型移动到 CPU 计算验证 loss，再保存 `final_checkpoint`。

### 5.4 测试优先的实施顺序

已按以下行为先增加自动化测试，再实现 `sft_mps.py`：

1. `TokenExtender` 从现有 index 得到 560 个唯一 token；
2. resize 后 tokenizer/model vocab size 一致；
3. `freeze_LLM=True` 时 Transformer 参数被冻结；
4. input embedding 和非绑定 output head 的新增 token 行可以获得梯度；
5. 旧 token 行梯度为 0；
6. MPS 配置不会启用 bf16、bitsandbytes 或 DDP；
7. sample 参数能限制三个训练子数据集和验证集。

定向测试和 CPU 验证测试共 8 项，dry-run shell 测试 1 项，项目全量回归结果为 `38 passed, 3 skipped`。实际训练仍应先执行一次 8-sample dry run，再执行正式 smoke。

检查产物：

```bash

test -f outputs/local_mps/sft-smoke/final_checkpoint/tokenizer.json
```

重新加载 checkpoint：

```bash
uv run python -c "from transformers import AutoModelForCausalLM, AutoTokenizer; p='outputs/local_mps/sft-smoke/final_checkpoint'; t=AutoTokenizer.from_pretrained(p); m=AutoModelForCausalLM.from_pretrained(p); print({'vocab':len(t),'model_vocab':m.get_input_embeddings().num_embeddings}); assert len(t)==m.get_input_embeddings().num_embeddings"
```

额外验收：

- `trainer_state.json` 中存在有限的 train/eval loss；
- final checkpoint 能在 CPU 上重载，避免把 MPS 设备状态写死；
- tokenizer 中能查到 index 的 560 个 SID token；
- 输出目录保留 `trainer_state.json` 和 `final_checkpoint`，不生成 MPS epoch 中间 checkpoint；
- 日志记录训练耗时和峰值内存。

### 5.8 停止条件与降级顺序

遇到以下情况立即停止：

- loss 为 NaN/Inf；
- MPS OOM 或系统 memory pressure 进入红色；
- 新增 SID token 数不是 560；
- model vocab 与 tokenizer vocab 不一致；
- checkpoint 无法重新加载。

若发生 MPS OOM，按以下顺序降级：

1. 将 cutoff length 从 192 依次降到 128、64；
2. 将 sample 从 32 降到 8；
3. 保持 `micro_batch_size=1`，不要增加 batch；
4. 确认 Transformer 主体确实被冻结；
5. 暂停 epoch 内评估，只在训练结束后验证；
6. 最后才将基础模型换成 135M 级 instruct model。

样本数量主要影响运行时间，不能显著降低模型权重本身占用；MPS OOM 时应优先检查 dtype、冻结状态、序列长度和重复模型加载。
